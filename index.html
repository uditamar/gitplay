What is Crossplane?

Control Planes

Build your own APIs and services with control planes that manage infrastructure declaratively.

Extend Kubernetes

Use Kubernetes to manage any resource anywhere - not just containers, but databases, networks, and more.

Component Library

Use a library of components to assemble your platform faster with pre-built integrations.

Getting Started

1

Install Crossplane

Install Crossplane on your Kubernetes cluster using Helm or kubectl.

kubectl create namespace crossplane-system
2

Install a Provider

Install a provider to manage resources from your cloud provider (AWS, GCP, Azure).

kubectl apply -f provider-aws.yaml
3

Create Resources

Define and create cloud resources using Kubernetes YAML manifests.

kubectl apply -f my-resources.yaml

Understanding Compositions

What are Compositions?

Compositions are the heart of Crossplane's platform engineering capabilities. They allow you to create higher-level abstractions by combining multiple cloud resources into a single, reusable API.

Composition Templates

Practical Example: Database with VPC

Let's create a composition that provisions a complete database setup with networking, security groups, and monitoring.

1. Define the XRD (Composite Resource Definition)

apiVersion: apiextensions.crossplane.io/v1 kind: CompositeResourceDefinition metadata: name: xdatabases.example.org spec: group: example.org versions: - name: v1alpha1 served: true referenceable: true schema: openAPIV3Schema: type: object properties: spec: type: object properties: region: type: string description: AWS region for the database engine: type: string enum: [mysql, postgresql] description: Database engine type instanceClass: type: string description: RDS instance class storageSize: type: integer description: Storage size in GB required: [region, engine, instanceClass, storageSize] claimNames: kind: Database plural: databases

2. Create the Composition

apiVersion: apiextensions.crossplane.io/v1 kind: Composition metadata: name: database-with-vpc spec: compositeTypeRef: apiVersion: example.org/v1alpha1 kind: XDatabase mode: Pipeline pipeline: - step: transform-vpc functionRef: name: function-go-templating input: inline: template: | apiVersion: ec2.aws.crossplane.io/v1beta1 kind: VPC metadata: name: {{ .observed.composite.resource.metadata.name }}-vpc spec: forProvider: region: {{ .observed.composite.resource.spec.region }} cidrBlock: "10.0.0.0/16" enableDnsHostnames: true enableDnsSupport: true writeConnectionSecretsToRef: name: {{ .observed.composite.resource.metadata.name }}-vpc-connection namespace: crossplane-system - step: transform-subnet functionRef: name: function-go-templating input: inline: template: | apiVersion: ec2.aws.crossplane.io/v1beta1 kind: Subnet metadata: name: {{ .observed.composite.resource.metadata.name }}-subnet spec: forProvider: region: {{ .observed.composite.resource.spec.region }} vpcIdRef: name: {{ .observed.composite.resource.metadata.name }}-vpc cidrBlock: "10.0.1.0/24" availabilityZone: {{ .observed.composite.resource.spec.region }}a - step: transform-security-group functionRef: name: function-go-templating input: inline: template: | apiVersion: ec2.aws.crossplane.io/v1beta1 kind: SecurityGroup metadata: name: {{ .observed.composite.resource.metadata.name }}-sg spec: forProvider: region: {{ .observed.composite.resource.spec.region }} vpcIdRef: name: {{ .observed.composite.resource.metadata.name }}-vpc description: "Security group for database" ingress: - fromPort: 3306 toPort: 3306 protocol: "tcp" cidrBlocks: ["10.0.0.0/16"] - step: transform-rds functionRef: name: function-go-templating input: inline: template: | apiVersion: rds.aws.crossplane.io/v1alpha1 kind: DBInstance metadata: name: {{ .observed.composite.resource.metadata.name }}-db spec: forProvider: region: {{ .observed.composite.resource.spec.region }} dbInstanceClass: {{ .observed.composite.resource.spec.instanceClass }} engine: {{ .observed.composite.resource.spec.engine }} allocatedStorage: {{ .observed.composite.resource.spec.storageSize }} vpcSecurityGroupIdRefs: - name: {{ .observed.composite.resource.metadata.name }}-sg dbSubnetGroupNameRef: name: {{ .observed.composite.resource.metadata.name }}-subnet-group masterUsername: admin masterUserPasswordSecretRef: name: {{ .observed.composite.resource.metadata.name }}-db-password key: password writeConnectionSecretsToRef: name: {{ .observed.composite.resource.metadata.name }}-db-connection namespace: crossplane-system

3. Use the Composition with a Claim

apiVersion: example.org/v1alpha1 kind: Database metadata: name: my-production-db namespace: default spec: region: us-west-2 engine: postgresql instanceClass: db.t3.micro storageSize: 20

Why Use Compositions?

Reusability

Define once, use everywhere. Create standardized infrastructure patterns for your organization.

Governance

Enforce security policies, compliance requirements, and best practices across all deployments.

Abstraction

Hide complexity from application teams while providing simple, consistent APIs.

Consistency

Ensure all environments follow the same patterns and configurations.

Advanced Composition Features

1

Function Pipeline

Chain multiple functions together to transform, validate, and enrich your resource definitions.

2

Environment-Specific Compositions

Create different compositions for dev, staging, and production environments with appropriate resource sizing.

3

Dynamic Resource Selection

Use functions to conditionally create resources based on input parameters or environment variables.
